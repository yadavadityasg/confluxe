## Problem

The Docker build uses `oven/bun:1.1` but your `bun.lock` was generated by a newer Bun (v1.2+, which introduced a new text lockfile format). Bun 1.1 can't parse it, falls back to a fresh resolve, and `--frozen-lockfile` fails. On EC2 this stalls/errors the build.

## Fix

Update `Dockerfile` to pin a Bun image that understands the current lockfile format, in both the `deps` and `build` stages.

Change:
```
FROM oven/bun:1.1 AS deps
...
FROM oven/bun:1.1 AS build
```
to:
```
FROM oven/bun:1.2 AS deps
...
FROM oven/bun:1.2 AS build
```

(Using the `1.2` tag tracks the latest 1.2.x patch. If you want a fully reproducible build, pin a specific digest later.)

## Why this is the right fix

- The error `Unknown lockfile version at bun.lock:2:22` is a hard signal of a Bun version mismatch, not a dependency problem.
- Your local machine (which generated `bun.lock`) is on Bun ≥ 1.2. Matching the container avoids re-resolving deps and keeps `--frozen-lockfile` meaningful (reproducible builds, no surprise upgrades on the server).
- No code, lockfile, or compose changes are needed.

## After applying

On the EC2 host:
```
docker compose build --no-cache
docker compose up -d
```

If you'd rather not bump Bun, the alternative is to delete `bun.lock` and let the container regenerate it — but you lose reproducibility, so bumping the image is preferred.
